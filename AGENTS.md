# AGENTS.md

이 문서는 이 저장소에서 작업하는 Codex 및 기타 코딩 에이전트가 따라야 할 기본 작업 규칙을 정의한다.

## 목표

- 사용자 요청을 해결하는 데 필요한 최소한의 변경만 수행한다.
- 수정 전에 관련 파일 1~3개와 직접 연결된 호출 지점을 먼저 읽고 코드 흐름을 이해한다.
- 요청이 명확하게 요구하지 않는 한 기존 동작을 유지한다.
- 사용자가 명시적으로 수정 또는 구현을 요청하지 않은 경우, 코드를 직접 변경하지 않는다.

## 기본 전제

- 이 저장소는 사용자가 계속 작업 중인 로컬 프로젝트라고 가정한다.
- 워킹 트리가 이미 변경된 상태일 수 있음을 전제로 한다.
- 현재 작업과 무관한 사용자 변경사항은 되돌리지 않는다.
- 작업에 꼭 필요하지 않으면 파일 이름 변경, 이동, 삭제를 하지 않는다.

## 프로젝트 위치

- 백엔드 앱 위치: `D:\alfo\mickle-muckle-api`

## 프로젝트 개요

- 프론트엔드 스택: Vite, React 19, TypeScript, React Router, TanStack Query, Tailwind CSS
- 경로 별칭: `@` 는 `src` 를 가리킨다.
- 개발 서버 기본 포트: `5173`
- 프론트엔드 엔트리 파일: `src/main.tsx`
- 앱 루트 및 라우팅 정의: `src/App.tsx`

## 주요 라우트

- `/` -> `src/pages/MonthPage/index.tsx`
- `/total` -> `src/pages/TotalPage/index.tsx`
- `/finance` -> `src/pages/FinancePage/index.tsx`

## 주요 디렉터리 구조

- `src/pages`: 라우트 단위 페이지 컴포넌트
- `src/components/layout`: 레이아웃 컴포넌트
- `src/components/customUi`: 도메인 성격이 있는 커스텀 UI 컴포넌트
- `src/components/ui`: 공용 UI 컴포넌트
- `src/hooks`: 데이터 조회 및 화면 로직 훅
- `src/api`: API 호출 모듈
- `src/types`: 도메인 타입 정의
- `src/utils`, `src/lib`: 공용 유틸리티

## 수정 원칙

- 사용자가 원인 분석, 설명, 리뷰만 요청한 경우에는 코드 수정 없이 답변을 우선한다.
- 코드 변경은 사용자가 명시적으로 수정, 구현, 리팩터링을 요청했을 때만 수행한다.
- 변경 범위는 작고 리뷰하기 쉽게 유지한다.
- 주변 코드의 스타일과 패턴을 우선적으로 따른다.
- 새로운 추상화를 추가하기 전에 기존 패턴을 재사용할 수 있는지 먼저 본다.
- 꼭 필요하지 않으면 여러 파일에 걸친 넓은 범위의 리팩터링은 피한다.
- 파일이 이미 유니코드를 사용하지 않는 한 ASCII를 우선한다.
- 주석은 코드만으로 의도가 충분히 드러나지 않는 경우에만 간단히 추가한다.

## 저장소 맞춤 작업 규칙

- 화면 수정 전에는 해당 페이지 파일과 함께 연결된 훅, API 호출부를 같이 확인한다.
- 공용 UI가 필요하면 새 컴포넌트를 만들기 전에 `src/components/ui` 또는 `src/components/customUi` 에 기존 패턴이 있는지 먼저 확인한다.
- 라우트 구조 변경이 필요하면 `src/App.tsx` 와 사이드바 같은 네비게이션 진입점도 함께 확인한다.
- 데이터 조회 로직을 수정할 때는 TanStack Query 사용 패턴을 우선 유지한다.
- API 스펙이 불명확하면 프론트엔드에서 임의로 맞추기보다 호출부와 타입 정의를 먼저 확인한다.

## 안전 원칙

- 사용자가 명시적으로 요청하지 않은 파괴적 git 명령은 실행하지 않는다.
- 작업 편의를 위해 사용자의 변경사항을 덮어쓰지 않는다.
- 예상하지 못한 로컬 변경이 현재 작업과 충돌하면 멈추고 확인한다.
- 저장소 밖 데이터를 변경하거나 권한 상승이 필요한 명령은 먼저 확인을 받는다.

## 작업 절차

1. 관련 파일이나 진입 지점을 찾는다.
2. 관련 파일과 직접 연결된 호출부, 상태 변경부, 렌더링 지점을 우선 읽는다.
3. 사용자 요청이 설명, 분석, 리뷰인지 아니면 실제 수정 요청인지 먼저 구분한다.
4. 수정이 필요한 경우 현재 문제를 해결하는 가장 작은 범위로만 변경한다.
5. 가능하면 변경 지점에 맞는 검증을 수행한다.
6. 무엇을 바꿨는지와 남아 있는 위험 요소를 정리한다.

## 코드 변경 선호사항

- 임시 우회보다 근본 원인 해결을 우선한다.
- 복잡하고 영리한 코드보다 명확하고 직접적인 코드를 선호한다.
- 꼭 필요하지 않으면 함수 시그니처와 컴포넌트 API를 유지한다.
- UI 작업에서는 기존 디자인 시스템과 상호작용 패턴을 존중한다.
- TypeScript에서는 추론이 불명확하거나 현재 오류가 나는 경우 명시적 타입을 추가한다.
- 기존 프로젝트에 이미 같은 문제를 해결한 예제가 있으면 그 패턴을 우선 재사용한다.

## 테스트 및 검증

- 가능한 범위에서 가장 작은 관련 검증을 우선 실행한다.
- 전체 검증이 무거우면 변경한 파일 또는 기능과 직접 관련된 검증만 수행한다.
- 검증 명령을 모르면 `package.json`, 설정 파일, 기존 문서를 먼저 확인한다.
- 검증을 실행하지 못했으면 최종 보고에서 그 사실을 분명히 밝힌다.
- 현재 프론트엔드에서 확인된 주요 명령은 아래와 같다.
  - `npm run dev`
  - `npm run build`
  - `npm run lint`
- 현재 `package.json` 기준 별도 `test` 스크립트는 없다. 테스트가 없으면 관련 범위의 `build` 또는 `lint` 를 최소 검증으로 사용한다.

## 커뮤니케이션

- 설명은 간결하고 사실 중심으로 작성한다.
- 저장소 맥락이 부족할 때는 가정을 분명히 적는다.
- 막히는 지점이 있으면 늦지 않게 공유한다.
- 리뷰 요청에서는 스타일보다 버그, 회귀 가능성, 테스트 누락을 우선적으로 본다.
- 사용자가 코드 관련 질문을 했을 때 현재 코드 스타일이 실무나 좋은 코드 컨벤션에 어긋나면, 수정된 코드나 예시 코드를 기존 스타일에 맞춰 제시하기 전에 더 권장되는 실무 기준과 방향을 먼저 안내한다.

- 이후 이 저장소에 스택별 또는 디렉터리별 세부 규칙이 추가되면 그 규칙이 이 기본 규칙보다 우선한다.
